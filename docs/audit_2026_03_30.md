# Field App Codebase & Data Integrity Audit — March 30, 2026

## Scope
Full audit of the Mandarte field app codebase and database, focused on data integrity and merge-readiness for eventual transfer to the final proofed database (breedfile + survival files).

## BUGS FIXED

### 0a. CRITICAL: Julian day off by 1 after DST spring-forward
**Files:** `lib/helpers.js` — `toJulianDay()`, `fromJulianDay()`, `estimateHatchDate()`, `calculateDFE()`
**Problem:** Used local-time `new Date(year, month-1, day)` for millisecond-based day counting. After DST spring-forward (March 8 in Pacific time), the hour lost makes the ms difference 1 hour short of a full day, causing `Math.floor()` to round down. Every Julian day from March 9 through December 31 was off by -1.
**Impact:** Every hatch date, DFE, banding/fledge/independence age warning calculated after March 8 would be wrong. On Mandarte, the entire breeding season (April-August) falls in this range.
**Fix:** Switched to `Date.UTC()` / `getUTCMonth()` / `getUTCDate()` for all Julian day conversions. DFE calculation simplified to pure integer arithmetic (Julian day subtraction) since both inputs and outputs are Julian days. Verified: all 366/365 days round-trip correctly for 2024/2025/2026.

### 0b. CRITICAL: Visit dates logged as tomorrow after 5pm PDT
**Files:** All 4 page.js files (9 instances total)
**Problem:** `new Date().toISOString().split('T')[0]` returns UTC date, not local date. After 5pm PDT (midnight UTC), this returns tomorrow's date. Similarly, visit time used `toTimeString().slice(0,5)` (local) while date used `toISOString` (UTC), creating date/time mismatches.
**Impact:** Any visit, nest observation, territory assignment, or bird addition done during evening fieldwork (common on Mandarte — work runs to 8-9pm in summer) would be stamped with the wrong date. This would corrupt DFE calculations, age estimates, and the entire visit timeline.
**Fix:** Created `localDateString()` and `localTimeString()` helpers that use `getFullYear()`/`getMonth()`/`getDate()` (local time). Replaced all 9 instances across all files. Added doc comment warning against `toISOString().split('T')[0]`.

### 1. CRITICAL: `parseInt(v) || null` converted 0 to null
**File:** `nests/[nestrec]/page.js` save handler
**Problem:** `parseInt("0") || null` evaluates to `null` because `0` is falsy in JavaScript. This meant if a researcher entered eggs=0, hatch=0, fledge=0 (all valid values for failed nests), the data would be silently discarded and the database field left unchanged.
**Fix:** Changed to `isNaN(parsed) ? null : parsed` which correctly preserves 0 values.

### 2. Negative band_id collision risk for unbanded birds
**File:** `birds/page.js` unbanded bird creation
**Problem:** `-(Date.now() % 1000000)` only produces 1M unique values. Two users creating birds in the same millisecond (or restarting the app) could collide.
**Fix:** Changed to `-(Date.now() * 1000 + Math.floor(Math.random() * 1000))` which uses the full timestamp plus randomness.

### 3. Territory card didn't show past males/females with date ranges
**File:** `territories/[code]/page.js`
**Problem:** Only fetched current assignments (end_date IS NULL). When a bird moved territories, the old bird disappeared from the card with no history.
**Fix:** Now fetches ALL assignments for the territory this year and displays a timeline showing each male and female with start → end dates. Current birds are highlighted with a blue/pink badge.

### 4. Nest link used `nestrec` which is NULL for field nests
**File:** `territories/[code]/page.js` nest list
**Problem:** `href={/nests/${nest.nestrec}}` — field-created nests have nestrec=NULL, so links broke.
**Fix:** Changed to `nest.nestrec || nest.breed_id` with appropriate display fallback.

## VERIFIED CORRECT

### Column Name Alignment (breed table)
Every field name in the card state matches the schema exactly. No mismatches found across all 40+ breed columns that the form touches.

### Integer vs Text Type Handling
- `cow_band` and `cow_fledge` are correctly kept as TEXT (not in intFields set). The schema defines them as TEXT to preserve uncertainty markers like "?" or "1+".
- All other numeric fields correctly use parseInt for the save.
- Quality flag fields (_quality suffix) correctly saved as TEXT.

### Field App Boundaries
- **survival table**: Read-only from app. Never inserted/updated. Correct — survival records are computed post-season.
- **proofed flag**: Never set to true by the app. Correct — proofing happens during data review.
- **orig_dfe**: Never touched by the app. Correct — pre-harmonization historical data only.
- **recruits**: Never touched by the app. Correct — computed after multi-year analysis.
- **raw_ tables**: Never touched by the app. Correct — immutable archive.

### Season Protection
Database triggers correctly block:
- Modification of previous-season records (year < current year)
- Modification of proofed records
- Deletion of any birds record
- Any modification to raw_survival and raw_breed (absolute block, no override)

### is_immigrant Handling
- Adult birds added via "returning" or "unbanded" modes: `is_immigrant` left as NULL. Correct per design — field app doesn't know yet.
- Chicks banded on nests (kid1-kid5): `is_immigrant = 0`. Correct — they hatched on Mandarte, definitively resident.
- Trigger prevents changing is_immigrant once set (0→1 or 1→0 blocked). Historical import fills NULLs.

### DFE Calculation Chain
- `date_hatch` auto-derived from chick age visits (highest reliability estimate preferred)
- DFE = hatch_date - 13 - (clutch_size - 1)
- Both are visible/editable on the nest card
- Only `dfe` is written; `corr_dfe` is reserved for 2014 data review corrections
- `orig_dfe` untouched

### Nest Parent Assignment
- Parents come from breed.male_id/female_id at nest creation time
- Looked up from territory_assignments at creation, then stored as foreign keys
- Territory changes after nest creation do NOT retroactively change parents. Correct per memory/design.

### study_year Calculation
`currentYear - 1974` → study_year 1 = 1975. Verified correct.

### nestrec Handling
- Field nests created with nestrec = NULL (correct — assigned during proofing)
- nest_visits link via breed_id (primary) and nestrec (fallback)
- Nest detail page loads by breed_id first, then falls back to nestrec

## LOW-RISK ISSUES (not fixed, documented for awareness)

### 1. Empty field doesn't clear database value
If you enter a value, save, then clear the field and save again, the old value persists. This is intentional (paper card model — you don't erase things), but could confuse users. The only way to "clear" a value would be direct SQL with admin override.

### 2. No range validation on Julian day fields
date_hatch and dfe accept any integer. A user could enter 999 or -1. Low risk since these are experienced researchers, but a range check (1-366) could be added.

### 3. nest_height migration conflict
migration_001 adds nest_height as TEXT, migration_007 tries to add it as DOUBLE PRECISION. Since ADD COLUMN IF NOT EXISTS means the first one wins, it's TEXT in practice. This is fine for now (text like "1.2m" works) but may need cleanup for analysis. Recommendation: keep as TEXT since historical data includes units.

### 4. No color combo format validation
Kid color combos are saved to birds.color_combo without validating format (e.g., "dbm.gr" pattern). Low risk since field workers know the format, but a helper could enforce it.

### 5. Duplicate migration_001 files
Both `migration_001_field_app.sql` and `migration_001_color_combos.sql` exist with overlapping content. Should be consolidated.

## MERGE-READINESS SUMMARY

The field app data is stored in a structure that maps cleanly to the final database:

| Field App Table | Maps To | Status |
|---|---|---|
| breed (working) | Final breedfile | Column names match exactly. nestrec assigned during proofing. |
| birds | Final birds/ninecodes | band_id is the join key. color_combo on birds table. |
| territory_assignments | Historical territory records | Timeline format with start/end dates. |
| nest_visits | New — field observation log | Links to breed via breed_id. No historical equivalent (paper cards weren't digitized visit-by-visit). |
| territory_visits | New — field observation log | Similar to nest_visits. |
| survival | Read-only | Field app never writes. Computed post-season. |
| corrections | Audit trail | Append-only. Documents all field corrections. |

**Key merge steps needed post-season:**
1. Assign nestrec values to field nests (sequential within year)
2. Fill in survival records from territory/nest observation data
3. Review and set proofed = TRUE on verified records
4. Import into final database structure alongside historical data
5. Resolve any import_conflicts via the staging system
